# Tử Vi Chart Schema Documentation

## Overview

This document describes the internal JSON schema for Tử Vi (Purple Star Astrology) charts generated by the system.

**Schema Version:** 1.0

---

## Chart Response Structure

```typescript
{
  "chart_type": "TUVI",
  "version": "1.0",
  "metadata": {
    "label": string,              // User-provided label or default
    "birth_date": string,         // ISO date (YYYY-MM-DD)
    "birth_time": string,         // Time (HH:MM or HH:MM:SS)
    "gender": string,             // "male" or "female"
    "calculated_at": string       // ISO timestamp of calculation
  },
  "palaces": {
    [palace_name: string]: {
      "name": string,             // Palace name (e.g., "Mệnh", "Phúc Đức")
      "stars": string[],          // List of star names in this palace
      "position": number | null,  // Palace position (1-12)
      "attributes": object        // Additional palace attributes
    }
  },
  "stars": {
    [star_name: string]: {
      "name": string,             // Star name
      "palace": string | null,    // Palace where star is located
      "brightness": string | null,// Star brightness/state (Miếu, Vượng, Đắc Địa, etc.)
      "category": string | null,  // Star category (Chính Tinh, Phụ Tinh, etc.)
      "attributes": object        // Additional star attributes
    }
  },
  "thien_can": string | null,     // Heavenly Stem
  "dia_chi": string | null,       // Earthly Branch
  "lunar_date": object | null,    // Lunar calendar date information
  "raw_data": object | null       // Raw output from lasotuvi (for debugging)
}
```

## Palace Names (12 Palaces)

The Tử Vi system uses 12 standard palaces (cung):

1. **Mệnh** (Life/Destiny Palace)
2. **Phụ Mẫu** (Parents Palace)
3. **Phúc Đức** (Fortune/Blessing Palace)
4. **Điền Trạch** (Property Palace)
5. **Quan Lộc** (Career Palace)
6. **Nô Bộc** (Servants/Friends Palace)
7. **Thiên Di** (Migration Palace)
8. **Tật Ách** (Health/Sickness Palace)
9. **Tài Bạch** (Wealth Palace)
10. **Tử Nữ** (Children Palace)
11. **Phu Thê** (Spouse Palace)
12. **Huynh Đệ** (Siblings Palace)

## Star Brightness Levels

Stars have different brightness levels indicating their strength:

- **Miếu** (Temple) - Strongest, most favorable
- **Vượng** (Prosperous) - Strong, favorable
- **Đắc Địa** (Grounded) - Moderate, stable
- **Bình Hòa** (Peaceful) - Neutral
- **Hãm** (Trapped) - Weak, challenged
- **Vô** (Empty/Void) - Weakest, least favorable

## Usage Notes

### For Frontend Developers

- Always check `chart_type` to determine which visualizer to use
- The `palaces` object contains the 12 palace grid for visualization
- The `stars` object contains star details for tooltips/details
- Use `metadata.label` for display purposes
- `raw_data` is for debugging only - do not rely on its structure

### For Backend Developers

- The schema is normalized to be consistent regardless of lasotuvi API changes
- Always populate `metadata.calculated_at` with UTC timestamp
- Keep `raw_data` for troubleshooting but don't expose it in production responses
- Handle missing/null fields gracefully

---

## Validation Rules

1. `birth_date` must be valid ISO date (YYYY-MM-DD)
2. `birth_time` must be in HH:MM or HH:MM:SS format
3. `gender` must be "male" or "female" (case-insensitive)
4. All 12 palaces should be present in response
5. Major stars (Chính Tinh) must be present in the `stars` object

---

## Error Responses

When chart calculation fails, the API returns:

```json
{
  "error": "Error message",
  "detail": "Detailed error description",
  "error_type": "InvalidDateError | CalculationError"
}
```

HTTP status codes:
- 400: Invalid input (bad date, time, or gender)
- 500: Calculation error (engine failure)

## Locked Contract for W2-ENGINE-04

This section is the source of truth for the create-chart flow between the UI,
chart engines, and Supabase `la_so` storage.

### UI Form Input

The chart creation form must collect and keep this canonical shape:

```typescript
type ChartSystem = "TUVI" | "BATU" | "TUVI_BATU";

interface CreateChartFormInput {
  label: string;
  birth_date: string;   // YYYY-MM-DD
  birth_time: string;   // HH:MM, 24-hour local time
  gender: "male" | "female";
  chart_system: ChartSystem;
}
```

Notes:

- The UI may display Vietnamese labels, but stored/submitted `gender` should be
  normalized to `"male"` or `"female"`.
- `BATU` and `TUVI_BATU` currently support years `1930-2048`, matching the
  local Bazi date mapping data.

### Engine Endpoints

Frontend create-chart flow must call these public endpoints:

```text
TUVI: POST <FASTAPI_BASE_URL>/chart/tuvi
BATU: POST /api/battu/calculate
```

`POST /lasotuvi/generate` remains a low-level diagnostic endpoint for the raw
lasotuvi wrapper. It should not be used by the create-chart UI.

`POST /chart/tuvi` request:

```json
{
  "label": "My chart",
  "birth_date": "1990-01-15",
  "birth_time": "14:30",
  "gender": "male"
}
```

`POST /api/battu/calculate` request:

```json
{
  "year": 1990,
  "month": 1,
  "day": 15,
  "hour": 14,
  "gender": "male",
  "label": "My chart"
}
```

The frontend adapter must derive Bát Tự `year`, `month`, `day`, and `hour` from
`birth_date` and `birth_time`.

### Supabase `la_so` Row

After successful engine calculation, insert exactly one row into `la_so`:

```typescript
interface LaSoInsert {
  user_id: string;
  label: string;
  birth_date: string; // YYYY-MM-DD
  birth_time: string; // HH:MM
  gender: "male" | "female";
  chart_system: "TUVI" | "BATU" | "TUVI_BATU";
  chart_data: TuViChartData | BatuChartData | TuViBatuChartData;
  chart_version: "1.0";
}
```

### `chart_data` Format

For a single-system chart, store the normalized engine response directly:

```typescript
// chart_system = "TUVI"
chart_data = TuViChartResponse;

// chart_system = "BATU"
chart_data = BatuChartResponse;
```

For combined charts, store both normalized responses under stable keys:

```json
{
  "tuvi": {
    "chart_type": "TUVI",
    "version": "1.0"
  },
  "batu": {
    "chart_type": "BATU",
    "version": "1.0"
  }
}
```

If either engine fails for `TUVI_BATU`, the create-chart flow should not insert
the `la_so` row. Show the engine error and let the user retry.

## Extended Tu Vi Rendering Fields

`POST /chart/tuvi` keeps the existing schema stable and adds optional rendering
metadata for the 12-palace board:

- `metadata.personal_info`: solar date, lunar date when available, birth hour
  branch, gender, yin/yang gender, and year/month/day/hour can-chi fields.
- `metadata.destiny_info`: birth-year nap am, cuc menh, chu menh, chu than,
  can luong, lai nhan cung, than cu, and Menh/Than palace indexes.
- `palaces[*].attributes.dia_chi`: palace earthly branch.
- `palaces[*].attributes.trang_sinh`: Trang Sinh cycle state.
- `palaces[*].attributes.dai_han_age`: displayed Dai Han age.
- `palaces[*].attributes.tieu_han`: Tieu Han marker.
- `palaces[*].attributes.luu_nien_dai_van`: annual major-cycle marker when the
  upstream engine provides one.
- `palaces[*].star_details`: stars as objects with `name`, `brightness`,
  `brightness_code`, `category`, `color`, and `id`.
- `palaces[*].star_groups`: stars grouped as `chinh_tinh`, `phu_tinh`, and
  `khac` so the frontend can render major stars first.

Brightness codes use `M`, `V`, `Đ`, `B`, and `H` for Mieu/Manh, Vuong, Dac,
Binh, and Ham. The frontend still accepts older charts that only contain
`palaces[*].stars` as strings.
